# Milestone 022 — Design the Intelligence Implementation Architecture

## Objective

Translate the approved methodology in `docs/VERDICT_INTELLIGENCE_FRAMEWORK.md`
into an implementation-ready architecture for Verdict's real evaluation
pipeline.

This milestone is **documentation only**.

Do **not** modify production code.

Do **not** implement the real AI engine.

Do **not** modify existing TypeScript interfaces.

The purpose of this milestone is to produce the architectural blueprint
that will guide every implementation milestone that follows.

---

# Background

Milestone 021 established **how Verdict should reason**.

It defined:

- the four evaluation dimensions
- evidence-first reasoning
- deterministic verdict computation
- deterministic confidence computation
- validation rules
- the AI contract

The methodology is now considered the product contract.

This milestone answers a different question:

> **How should this methodology be implemented inside the existing codebase?**

The output should bridge today's deterministic mock engine and tomorrow's
real model-backed evaluation engine without introducing unnecessary
complexity.

# Guiding Principles

Throughout this milestone:

- Prefer simplicity over sophistication.
- Every abstraction must justify its existence.
- Prefer deterministic software over model judgment whenever possible.
- AI should observe.
- Server-side code should decide.
- Every verdict should be explainable.
- Every architectural recommendation should improve maintainability.
- Avoid designing for hypothetical future requirements.

Whenever there is a trade-off between a clever architecture and an obvious
architecture, prefer the obvious one.

Describe the entire flow from:

Creative submission

↓

Image preprocessing

↓

CreativeContext construction

↓

Model request

↓

Structured model response

↓

Validation

↓

Evidence extraction

↓

Finding generation

↓

Severity validation

↓

Deterministic verdict computation

↓

Deterministic confidence computation

↓

Recommendation generation

↓

Executive summary generation

↓

VerdictReport

↓

Persistence

↓

UI rendering

Clearly identify:

- which stages belong to AI
- which belong to deterministic application logic
- which belong to the presentation layer

---

## 2. Define architectural boundaries

Determine the responsibilities of:

- mock engine
- future OpenAI engine
- prompt construction
- structured output parser
- validation layer
- deterministic decision engine
- report assembler
- persistence layer

For every responsibility explain:

- what it owns
- what it should never own
- its inputs
- its outputs

Avoid unnecessary abstractions.

---

## 3. Review the domain model

Review:

- VerdictReport
- AnnotatedPoint
- Weakness
- CreativeContext
- VerdictEngine

Recommend:

- additions
- removals
- deprecations
- renamed concepts
- internal-only types
- public API types

Do not modify code.

Instead, include proposed interfaces where useful.

---

## 4. Design the AI contract

Specify exactly what the model should return.

For every field classify it as one of:

- AI-generated
- server-computed
- validated
- derived

Determine whether the model should generate:

- findings
- evidence
- recommendations
- executive summary
- verdict
- confidence

If a field should no longer be generated by AI,
explain why.

---

## 5. Validation layer

Design deterministic validation rules for:

- missing evidence
- invalid categories
- illegal blocking values
- contradictory findings
- malformed bounding boxes
- duplicate findings
- inconsistent recommendations

Determine:

- retry conditions
- coercion rules
- rejection rules

---

## 6. Decision engine

Translate the methodology into implementation-ready software.

Specify:

- proposed pure functions
- function responsibilities
- function inputs
- function outputs

The implementation should remain fully deterministic.

---

## 7. Migration strategy

Describe the safest migration from:

mock-engine

↓

hybrid engine

↓

real evaluation engine

The migration should minimize disruption to the existing UI.

---

## 8. Testing strategy

Recommend how each layer should be tested independently.

Include:

- unit tests
- deterministic regression tests
- schema validation tests
- prompt validation
- integration tests

The architecture should maximize testability.

---

## 9. Risks

Identify implementation risks including:

- unnecessary coupling
- over-engineering
- prompt brittleness
- validation complexity
- hidden business logic inside prompts
- future maintenance burden

Recommend mitigation strategies.
